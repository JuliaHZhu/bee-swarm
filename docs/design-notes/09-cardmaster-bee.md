# Cardmaster Bee — 战术本 + 参谋长

> *战术本：每种局面该出什么招，标的物是什么。参谋长：不决策，只做参谋——复盘、解读、建议。*

---

## 一、核心隐喻：回合抽卡

```
┌─────────────────────────────────────────┐
│              你的回合                     │
│                                         │
│  抽卡 → 动作类型 + 目标客体               │
│    │                                     │
│    ├─ 客体是人？→ 说服 or 交易            │
│    └─ 客体是事？→ 生产 or 研究            │
│         │                                │
│         ▼                                │
│  关键标的物是什么？                        │
│    ├─ 说服 → report                      │
│    ├─ 交易 → product demo                │
│    ├─ 生产 → pipeline                    │
│    └─ 研究 → standard                    │
│         │                                │
│         ▼                                │
│  调度蜂群生产标的物 → 标的物质量 → 成功率   │
│         │                                │
│         ▼                                │
│  结算：成功/失败 → 下一回合               │
└─────────────────────────────────────────┘
```

## 二、四象限动作矩阵

Cardmaster 只有 **4 种动作**。每一种都有明确的目标、关键标的物、标的物生产者、成功率判定。

| | 客体：人 | 客体：事 |
|---|---|---|
| **目的：接触/绑定** | **说服** (Persuade) | **生产** (Produce) |
| **目的：获取/提取** | **交易** (Trade) | **研究** (Research) |

### 象限详解

#### 说服 (Persuade) — 客体是人，把他绑在信息链上

| 字段 | 内容 |
|------|------|
| **目标** | 让对方接受你的信息框架，进入你的信息链 |
| **关键标的物** | **report** — 一份让对方无法拒绝的论证报告 |
| **标的物内容** | 你为什么找他、你的信用证据、他能得到什么、不加入的代价 |
| **标的物生产者** | Worker Bee（调研+写作），Verification Bee（事实校验） |
| **成功率公式** | `report质量 × 对方关系基础 × 时机` |
| **失败处理** | 回到研究动作——report 不够好是因为标准不对 |

```
例:
  抽卡: 说服 → 目标: 投资人张总
  标的物: 一份市场分析 report（证明你的方向是唯一正确的）
  调度: Chef Bee → 拆 task → Worker 调研+写作 → World 验证数据
  结算: report 通过 World 验证 → 成功率 85% → 发给他
```

**report 质量评估维度**（Verification Bee 验证后打分）:
1. 数据准确度（triangulated claims 占比）
2. 逻辑完整性（证据链是否有断裂）
3. 针对性（是否回应了对方已知的顾虑）
4. 信用背书（引用的来源可信度）

#### 交易 (Trade) — 客体是人，获得他的资源

| 字段 | 内容 |
|------|------|
| **目标** | 用你有的换他没有的（钱、时间、渠道、数据、背书） |
| **关键标的物** | **product demo** — 让对方看到"这东西已经能跑了" |
| **标的物内容** | 可运行的原型、可测量的指标、可对比的竞品分析 |
| **标的物生产者** | Worker Bee（开发+测试），Verification Bee（性能基准验证） |
| **成功率公式** | `demo完成度 × 对方需求匹配度 × 替代方案稀缺性` |
| **失败处理** | 回到生产动作——demo 不够好是因为流水线有问题 |

```
例:
  抽卡: 交易 → 目标: 渠道商李总
  标的物: product demo（你的产品在他的渠道跑一遍的数据）
  调度: Chef Bee → 拆 task → Worker 开发 demo → World 验证性能
  结算: demo 通过性能基准 → 成功率 72% → 约演示时间
```

**demo 质量评估维度**:
1. 功能完整度（承诺的功能实现了多少）
2. 性能指标（和竞品的对比数据）
3. 稳定性（Verification Bee 压测结果）
4. 适配度（在对方环境/场景下的表现）

#### 生产 (Produce) — 客体是事，build 流水线

| 字段 | 内容 |
|------|------|
| **目标** | 建立可重复的生产能力 |
| **动作形式** | **build 流水线** |
| **产品** | **pipeline** — 一套标准化的生产流程 |
| **产品内容** | 输入规范 → 处理步骤 → 输出规格 → 质检标准 → 扩展点 |
| **产品生产者** | Skeleton Bee（设计流水线结构）+ Chef Bee（拆 task）+ Worker Bee（实现） |
| **成功标准** | 流水线跑通 3 次，产出方差 < 阈值 |
| **失败处理** | 回到研究动作——流水线卡住是因为标准没定清楚 |

```
例:
  抽卡: 生产 → 目标: 论文写作流水线
  产品: pipeline（文献调研→大纲→初稿→修改→定稿 五阶段流水线）
  调度: Skeleton Bee → 设计流水线结构 → Chef Bee → 拆 task → Worker 实现
  结算: 流水线产出 3 篇论文片段，质量方差 < 0.1 → 成功
```

**pipeline 质量评估维度**:
1. 可重复性（跑 3 次产出质量是否稳定）
2. 容错性（某一步失败后能否自动恢复）
3. 可扩展性（新增步骤是否只需修改配置）
4. 效率（端到端耗时）

#### 研究 (Research) — 客体是事，build 标准

| 字段 | 内容 |
|------|------|
| **目标** | 建立判断对错的标尺 |
| **动作形式** | **build 标准** |
| **产品** | **standard** — 一套可操作的判断标准 |
| **产品内容** | 定义 → 测量方法 → 阈值 → 边界案例 → 更新机制 |
| **产品生产者** | Aristotle Bee（术语定义）+ Skeleton Bee（标准结构）+ Verification Bee（数据验证） |
| **成功标准** | 标准被 ≥2 个不同的 Worker 任务正确使用，判断结果一致 |
| **失败处理** | 再研究——标准不够清晰是因为定义还没对齐 |

```
例:
  抽卡: 研究 → 目标: "什么是一篇好论文"
  产品: standard（评估论文的 5 维度标准）
  调度: Aristotle Bee → 定义"好"的每个维度 → Skeleton Bee → 标准结构 → Verification Bee 验证
  结算: 标准被 3 个 Worker 任务使用，评分一致 → 成功
```

**standard 质量评估维度**:
1. 可操作性（能否不依赖主观判断直接使用）
2. 一致性（不同人/Worker 用同一标准得出相同结论）
3. 覆盖率（标准覆盖了所有已知边界案例）
4. 可更新性（新数据出现后能否自动修正）

## 三、回合制机制

### 卡组

卡组 = 当前可用的动作集合。初始卡组包含全部 4 种动作。随着游戏进行，卡组可以扩展（新的子类型）。

```json
{
  "deck": [
    {"action": "persuade",  "target_type": "person", "artifact": "report"},
    {"action": "trade",     "target_type": "person", "artifact": "product_demo"},
    {"action": "produce",   "target_type": "thing",  "artifact": "pipeline"},
    {"action": "research",  "target_type": "thing",  "artifact": "standard"}
  ],
  "hand_size": 1,
  "discard": []
}
```

### 回合流程

```
回合开始
  │
  ├─ 1. 抽卡: 从卡组抽 1 张（或人指定"这回合我要说服"）
  │
  ├─ 2. 指定目标: 人 → 具体是谁？事 → 具体是什么？
  │
  ├─ 3. 标的物需求: Cardmaster 生成标的物规格书
  │     ├─ 说服张总 → 需要一份关于 X 市场的 report
  │     ├─ 交易李总 → 需要一个在 Y 平台的 product demo
  │     ├─ 生产论文流水线 → 需要一条 5 阶段 pipeline
  │     └─ 研究"好论文标准" → 需要一套 5 维度 standard
  │
  ├─ 4. 调度蜂群: Chef Bee 拆 task → Workers 执行 → Verification Bee 验证
  │     （这一步可能跨越多轮 Worker 执行）
  │
  ├─ 5. 结算: 标的物质量 → 成功率 → 发 or 不发
  │     ├─ 质量达标 → 发送（说服/交易）or 部署（生产/研究）
  │     └─ 质量不达标 → 分析失败原因 → 回到对应动作
  │           ├─ report 不好 → 研究标准有问题 → 下回合抽"研究"
  │           ├─ demo 不好 → 生产流水线有问题 → 下回合抽"生产"
  │           ├─ pipeline 不好 → 研究标准没定清楚 → 下回合抽"研究"
  │           └─ standard 不好 → 再研究
  │
  └─ 6. 下一回合
```

### 动作之间的转换图

```
           report 不好 → 研究
              ↑            │
           说服 ←──────────┘
              │
    demo 不好 → 生产
              ↑            │
           交易 ←──────────┘
              │
 pipeline 不好 → 研究
              ↑            │
           生产 ←──────────┘
              │
standard 不好 → 再研究
              ↑            │
           研究 ←──────────┘
```

**核心循环**: 说服/交易 依赖 报告/demo → 报告/demo 不好 → 回到 研究/生产 补课 → 再试。

这不是线性的——是**螺旋上升的**。每次失败都会让标准更清晰、流水线更稳定。

## 四、标的物规格书

Cardmaster 抽卡后，生成一份标的物规格书，交给 Chef Bee 执行。这是 Cardmaster 的核心产出。

```markdown
# 标的物规格书: report-2026-0701-001

## Meta
- 动作类型: persuade（说服）
- 目标客体: 投资人张总（人）
- 回合: R3
- 前置回合: R1（研究"好report标准"）, R2（生产 report 流水线）

## 目标 Profile
- 姓名: 张总
- 角色: 天使投资人
- 已知关注点: AI 基础设施、开源商业模式
- 已知顾虑: "开源怎么赚钱？"
- 关系基础: 一面之缘（2026-06 行业会议）

## 标的物需求
- 类型: 市场分析 report
- 核心论证: "开源 AI 基础设施是下一个 10 年的基础设施层，先发优势不可逆"
- 必需章节:
  1. 市场规模（TAM/SAM/SOM）— 数据来源: 公开报告 + 自行测算
  2. 竞品格局 — 谁在做、我们和他们的区别
  3. 开源商业模式验证 — 成功案例（Red Hat, MongoDB, GitLab）
  4. 团队信用 — 为什么是我们
  5. 风险与缓解 — 诚实列出风险 + 应对方案
- 质量要求:
  - 数据准确度: triangulated ≥ 60%
  - 逻辑完整性: 证据链无断裂
  - 针对性: 明确回应张总的顾虑（开源怎么赚钱）

## 调度
- Chef Bee: 拆成 5 个 task（每章 1 个 + 交叉验证 1 个）
- Worker: 5 个 Worker 并行调研+写作
- Verification Bee: 验证所有数据 claim，产出证据链报告
```

## 五、Cardmaster 与蜂群的接口

```
                 Strategy Bee（长期方向）
                       │
                       │ "需要融资" or "需要拓展渠道"
                       ▼
                 Cardmaster Bee
                       │
        ┌──────────────┼──────────────┐
        │              │              │
        ▼              ▼              ▼
    Chef Bee       Architecture    Aristotle
   （拆task）     Bee（结构）    Bee（定义）
        │              │              │
        └──────────────┼──────────────┘
                       │
                       ▼
                  Worker Bees
                       │
                       ▼
                  Verification Bee（验证）
                       │
                       ▼
                 Cardmaster Bee（结算）
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
      发出去        不达标        下一回合
    （目标达成）   （分析原因）   （抽新卡）
```

**Cardmaster 不执行**——它只做 4 件事:
1. 决定这回合做什么动作
2. 写标的物规格书
3. 调度其他 Bee
4. 结算成功率

## 七、Cardmaster 的 6 个 Skill

| # | Skill | 功能 | 调 LLM |
|---|-------|------|--------|
| 1 | **action-select** | 分析当前状态 → 推荐本回合动作（or 人指定） | 是 |
| 2 | **target-profile** | 为目标客体（人/事）生成画像：已知信息、顾虑、关系基础 | 是 |
| 3 | **spec-generate** | 根据动作类型 + 目标画像 → 生成标的物规格书 | 是 |
| 4 | **spec-dispatch** | 将规格书拆成调度指令 → Chef Bee（拆 task）+ 指定验证者（Verification Bee） | 否（纯规则） |
| 5 | **quality-evaluate** | 读取 Verification Bee 验证结果 → 标的物质量评分 → 成功率 | 否（纯规则） |
| 6 | **retro-analyze** | 失败时分析原因 → 推荐下一回合动作（回到哪个象限补课） | 是 |

### [1] action-select — 选择本回合动作

| 字段 | 内容 |
|------|------|
| **Input** | 上一回合的结果 + Strategy Bee 方向 + 卡组状态 |
| **Output** | 本回合动作: {action_type, target} |
| **调 LLM** | 是 |

**决策辅助**（LLM 建议，人确认）:
```
当前状态:
  - 上一回合: 研究"好 report 标准" → 成功（标准已建立）
  - Strategy Bee 方向: 需要融资
  - 卡组: 4 张全可用

推荐: 说服 → 投资人张总（有标准，可以做 report 了）
备选: 交易 → 渠道商李总（和融资并行？但先拿到钱更重要）
```

### [2] target-profile — 目标画像

| 字段 | 内容 |
|------|------|
| **Input** | 目标客体名称 + 类型（人/事） |
| **Output** | 目标画像: 已知信息、顾虑、关系基础、预期难度 |
| **调 LLM** | 是 |

### [3] spec-generate — 标的物规格书

| 字段 | 内容 |
|------|------|
| **Input** | 动作类型 + 目标画像 + 已建立的标准/流水线 |
| **Output** | 完整标的物规格书（格式见第四节） |
| **调 LLM** | 是 |

### [4] spec-dispatch — 调度蜂群

| 字段 | 内容 |
|------|------|
| **Input** | 标的物规格书 |
| **Output** | Chef Bee 的 GOAL.md（交给 Chef Bee 做行动计划） |
| **调 LLM** | 否（纯规则：把规格书的章节 → Chef Bee 的 task 骨架） |

**转换规则**:
```
report 规格书的 5 个必需章节
  → Chef Bee GOAL.md: 5 个 task + 1 个交叉验证 task
  → Chef Bee 自动进入 plan-generate 流水线

demo 规格书的功能清单
  → Chef Bee GOAL.md: 功能 task + 性能 task + 适配 task

pipeline 规格书的阶段定义
  → Skeleton Bee（设计结构）→ Chef Bee（每个阶段拆 task）

standard 规格书的维度定义
  → Aristotle Bee（定义术语）→ Skeleton Bee（标准结构）
```

### [5] quality-evaluate — 质量评估

| 字段 | 内容 |
|------|------|
| **Input** | Verification Bee 验证结果（证据链报告） + 标的物规格书 |
| **Output** | 质量评分（0-100）+ 成功率（%）+ 发/不发 |
| **调 LLM** | 否（纯规则） |

**评分公式**:
```
report 质量 = (
  triangulated_ratio × 40 +    # 验证通过的数据占比
  evidence_chain_score × 30 +   # 证据链完整性
  target_relevance × 30         # 对目标的针对性
) / 100

成功率 = quality × relationship_factor × timing_factor
  relationship_factor: 陌生人 0.5 / 一面之缘 0.7 / 熟人 0.9
  timing_factor: 紧急窗口 1.2 / 正常 1.0 / 淡季 0.8

阈值: 成功率 ≥ 70% → 发
      成功率 < 70% → 不达标 → retro-analyze
```

### [6] retro-analyze — 失败复盘

| 字段 | 内容 |
|------|------|
| **Input** | 失败的回合日志 + quality-evaluate 得分明细 |
| **Output** | 失败原因 + 推荐下一回合动作 |
| **调 LLM** | 是 |

**推荐规则**:
```
report 的 triangulated_ratio 低 → 数据源不够 → 下一回合: 研究（更好的数据标准）
report 的 target_relevance 低 → 不了解目标 → 下一回合: 研究（目标画像标准）
demo 的功能完整度低 → 开发能力不够 → 下一回合: 生产（更好的开发流水线）
demo 的性能指标低 → 技术基础不够 → 下一回合: 生产（优化流水线）
pipeline 的可重复性低 → 流程不稳定 → 下一回合: 研究（更好的质量标准）
standard 的一致性低 → 定义不清晰 → 下一回合: 研究（再次对齐定义）
```

## 八、设计原则

1. **四种动作就够了** — 不要加第五种。世界的复杂度由卡组扩展（子类型）承载，不增加基础动作。
2. **标的物驱动** — 每一回合只有一个关键交付物。不扩散。
3. **蜂群是工厂** — Cardmaster 不自己生产标的物。它写规格书，蜂群生产。
4. **螺旋上升** — 失败不是倒退，是让标准更清晰的机会。
5. **回合之间不耦合** — 每回合独立结算。上回合的失败只影响下回合的动作选择，不影响上回合的结算。

## 九、第 10 步：博弈复盘（外部反馈解读）

### 在十步流程中的位置

| 步骤 | 参与者 | Cardmaster 角色 |
|------|--------|----------------|
| 10 博弈复盘 | 人 + Cardmaster | 外部反馈解读 + 打法优化 |

### 前置条件
项目交付完成（第 7 步补丁收敛 + 第 9 步战役报告）。

### 输入
- 发表/合作/人际接触等外部反馈结果
- 战役报告（第 9 步，了解"做了什么"）
- 历次 Cardmaster 回合日志（了解"怎么打的"）

### 核心工作

#### 1. 我是不是做对了？
把外部反馈对照当初的标的物和动作选择，逐回合复盘：
- 说服动作 → 对方的实际反应是什么？report 质量够不够？
- 交易动作 → 成交了吗？demo 打动对方了吗？
- 生产动作 → 流水线经受住外部检验了吗？
- 研究动作 → 标准在市场/学术/合作中成立了吗？

#### 2. 对面是什么意思？
解读外部反馈的潜台词：
- 对方说"再看看" → 是 report 说服力不够还是时机不对？
- 对方说"不错但..." → 是 demo 功能缺失还是需求不匹配？
- 外部批评/拒绝 → 是方向问题还是执行问题？

#### 3. 怎么打可能能让我达到目的？
基于复盘提出调整建议：
- 先补研究（标准不够清楚）再打说服？
- 先补生产（demo 不够硬）再打交易？
- 换目标客体？（这个人不对，换赛道）
