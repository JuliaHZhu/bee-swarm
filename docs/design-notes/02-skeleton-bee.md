# Skeleton Bee — 骨架蜂

> *架构在脑袋里是将出未出到逐渐清晰化的过程。*
> *目标：规约到不能规约。算法是效率和资源的量纲。*

## 定位

Skeleton Bee 只在**选牌型阶段**（十步流程第 2 步）参与工作。与 human + Strategy Bee 协作，把模糊的战略目标翻译成具体的产出形态和抽象验收标准。

项目进入第 3 步后，Skeleton Bee 退场。

## 在第 2 步中的角色

当 human 和 Strategy Bee 确定"这个问题值得做"后，Skeleton Bee 介入帮助回答：

1. **产出形态**：这个项目最终是什么？产品？服务？报告？流水线？标准？
2. **验收标准**：几个关键抽象问题，必须能回答才算做完
3. **结构骨架**：产出物的核心模块/阶段是什么，怎么组合

这就是"选牌型"——不同任务类型对应不同产出形态。运营任务运营什么、写作任务写作什么，各不一样。Skeleton Bee 帮助把这种差异结构化成可操作的形式。

## 核心哲学

### 1. 规约到不能规约（Reduction to Irreducible）

架构一开始是宏观的、混沌的。要让它清晰化，必须一层一层往下剥——**rank / 降维 / 规约**，直到露出那个再也拆不动的核心约束。

### 2. 正交基底（Orthogonal Basis）

核心暴露后，结构必须是**高内聚、低耦合**的：
- 每个模块/组件 = 一个基底向量，只处理一个维度
- 无重叠 —— 两个组件共享责任 = 基底不正交
- 完整覆盖 —— 每个需求都有人负责
- 可组合 —— 通过接口/衔接点组合

### 3. Big O 作为资源量纲

Archetype（原型）暴露后，要考虑"怎么实现"。算法选择的标准就是**效率与资源**：
- 软件：时间复杂度、空间复杂度
- 内容：观众认知负荷、场景切换成本、时长预算
- 书籍：概念密度、阅读成本、认知递进梯度

## Skill 流水线（5 阶段）

通过**外源信息素 Markdown 文件**接力。

```
┌─────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│  1. capture-     │     │  2. decompose-   │     │  3. reduce-to-   │
│     intent       │────▶│      goal        │────▶│      core        │
│  捕捉原始冲动     │     │   目标展开分解    │     │  规约到不能规约   │
└─────────────────┘     └──────────────────┘     └──────────────────┘
                                                          │
┌─────────────────┐     ┌──────────────────┐               │
│ 5. evaluate-    │◀────│ 4. expose-       │◀──────────────┘
│   complexity    │     │    archetype     │
│  Big O 资源评估  │     │   暴露结构原型   │
└─────────────────┘     └──────────────────┘
```

### Stage 1: capture-intent — 捕捉原始冲动

- **输入**: Strategy Bee 的战略报告 + human 的"我想做 X"
- **产出**: `~/.worker-bee/arch/<project>/01-intent.md`
- **行为**: 把模糊的冲动锚定成文字

### Stage 2: decompose-goal — 目标展开

- **输入**: `01-intent.md`
- **产出**: `~/.worker-bee/arch/<project>/02-goals.md`
- **行为**: 拆成层级结构。每个子目标必须回答"为什么必须存在"。划定反目标（Anti-goals）。

### Stage 3: reduce-to-core — 规约到不能规约（🔥 灵魂）

- **输入**: `02-goals.md`
- **产出**: `~/.worker-bee/arch/<project>/03-core.md`
- **行为**: 对每个子目标不断问 **"为什么"**，直到答案变成物理现实或用户刚需。用**移除测试**确认。

### Stage 4: expose-archetype — 原型暴露

- **输入**: `03-core.md`
- **产出**: `~/.worker-bee/arch/<project>/04-archetype.md`
- **行为**: 把核心约束翻译成结构原型（产出形态）。引入正交基底检验。

### Stage 5: evaluate-complexity — 复杂度/资源评估

- **输入**: `04-archetype.md`
- **产出**: `~/.worker-bee/arch/<project>/05-complexity.md`
- **行为**: 对每条实现路径做效率与资源的量能评估，给出取舍理由。

## 外源信息素接力格式

```
~/.worker-bee/arch/
└── <project>/
    ├── 01-intent.md      ← capture-intent
    ├── 02-goals.md       ← decompose-goal
    ├── 03-core.md        ← reduce-to-core
    ├── 04-archetype.md   ← expose-archetype
    └── 05-complexity.md  ← evaluate-complexity
```

## 设计决策

| 决策 | 理由 |
|------|------|
| 拆成 5 个 skill，而非 1 个 | 小马过河：问题简单走到 Step 2 就够，问题深一路走到 Step 5 |
| 文件接力，非内存状态 | 外源信息素原则：人在回路，随时可改 |
| 反目标（Anti-goals） | 防止范围蔓延，不做什么和做什么同等重要 |
| 移除测试 | 核心约束的判定标准：去掉它，目标还成立吗？ |
| Big O 泛化 | 不仅软件，任何架构都有"资源与效率"的量纲 |
