一、定位与角色

1.1 Worker-Bee 在蜂群中的位置

🔮 PM Bee — 想法 → 方案

↓ 产出规划文档 / 任务拆解方案
⚔️ Centurion Bee — 分活 + 调度

↓ 派发任务卡片
🐝 Worker Bee — 执行 + 产出

Worker-Bee 是蜂群的执行单元。它不思考战略、不拆分解任务，只做一件事：从任务池里领取任务卡片，用手里的工具把活干出来，把产出物放回约定的位置。

1.2 核心设计原则

原则 1：任务即契约。Worker-Bee 只认任务卡片。卡片上写清楚了要什么、怎么验收，Worker 就照着干。没写清楚的，要么不接，要么往上退。
原则 2：File-as-Bus。所有输入输出都走文件系统。没有消息队列、没有 RPC、没有共享内存。命名规范就是协议。
原则 3：半异构同底座。所有 Worker 共享同一套 agent 循环 + 工具框架 + 记忆系统，差异只在技能包（skill pack）和配置。
原则 4：失败比成本易检测。优先设计快速失败机制。任务干砸了就标失败，让 Centurion 重试或重派，不要硬撑着凑结果。

1.3 Worker-Bee 的职责边界

✅ Worker 该做的❌ Worker 不该做的
• 从任务池领取任务卡片
• 理解任务目标和验收标准
• 调用工具执行任务
• 管理自己的工作记忆
• 产出符合验收标准的结果文件
• 报告完成状态（成功/失败/阻塞）
• 更新自己的本地知识图 		• 拆任务（那是 Centurion 的活）
• 规划大方向（那是 PM 的活）
• 决定任务优先级
• 给其他 Worker 派活
• 自己给自己加任务
• 跨项目协调资源
• 审查自己的产出（有 Review Bee）

三、nanobot 底座复用方案

3.1 nanobot 核心架构回顾

nanobot 是极简 agent 框架，核心约 4000 行 Python，MIT 协议。核心特点：

  • 极简 agent 循环：think → act → observe 三环，没有花活
  • MCP 原生支持：直接接入任何 MCP server 的工具
  • 多模型后端：OpenAI / Anthropic / Gemini / 本地模型 等全支持
  • 多平台接入：CLI / Discord / Telegram / Slack / 微信 等
  • 插件系统：skill 以 Python 模块形式热加载

3.2 复用清单

✅ 直接复用（代码级）

Tool Calling 框架（复用率 90%）
  • 工具定义装饰器和注册机制
  • 函数调用的参数解析和校验
  • 工具执行和结果回注
  • 错误处理和重试
MCP Client（复用率 95%）
  • MCP server 的启动/通信/工具发现
  • stdio 传输层
  • 工具结果的标准化处理
LLM Backend 适配层（复用率 95%）
  • 多模型后端的统一接口抽象
  • 消息格式转换（各家 API 的消息 schema 不同）
  • 流式输出支持
  • 配置管理（API key、模型名、温度等）

🔧 轻量改造（结构复用，逻辑修改）

Agent 主循环（改造率 ~30%）
  • nanobot 的循环是「用户发消息 → agent 回复」的对话模式
  • 改成「领取任务卡片 → 执行任务 → 产出结果 → 报告状态」的任务模式
  • 核心的 think→act→observe 三环不变
  • 加入：任务卡片解析、验收标准检查、产出物写入、状态上报
会话管理（改造率 ~50%）
  • nanobot 的会话是「用户 - agent」对话会话
  • 改成「任务 - worker」执行会话
  • 每个任务一个独立的上下文窗口
  • 会话结束（任务完成/失败）后，记忆归档到长期知识库

❌ 不用（完全替换）

平台接入层（Discord / Telegram / 微信 等）
  • Worker-Bee 不需要和人类用户直接对话
  • 所有交互通过 File-as-Bus 完成
  • 但保留 CLI 模式用于调试
nanobot 的原始插件集
  • 面向聊天场景的插件（表情包、查天气等）不需要
  • 通用工具（文件读写、shell 执行、web 搜索）保留
  • 用蜂群自己的技能包机制替代

3.3 代码提取策略

不是 fork 整个 nanobot 仓库过来改，而是提取核心模块 + 封装为 bee 底座：

src/base/
├── agent_loop.py       # 从 nanobot.core.agent 提取 + 改造
├── tool_manager.py     # 从 nanobot.core.tools 提取
├── mcp_client.py       # 从 nanobot.core.mcp 提取（基本不动）
├── llm_backend.py      # 从 nanobot.core.llm 提取（基本不动）
├── config.py           # 从 nanobot.core.config 提取 + 蜂群配置扩展
└── exceptions.py       # 从 nanobot.core.exceptions 提取

保留 nanobot 的 MIT 版权声明，在每个提取文件头部注明来源。

四、Agent 循环与决策机制

4.1 任务驱动型循环

nanobot 原生是对话驱动循环（user message → agent response），Worker-Bee 改成任务驱动循环：

┌─────────┐     ┌─────────────┐     ┌──────────────┐
│ 领取任务 │────▶│ 理解任务目标 │────▶│ 制定执行计划 │
└─────────┘     └─────────────┘     └──────┬───────┘
                                           │
                                    ┌──────▼───────┐
                                    │  选择工具     │
                                    └──────┬───────┘
                                           │
                                    ┌──────▼───────┐
                                    │  执行工具     │
                                    └──────┬───────┘
                                           │
                                    ┌──────▼───────┐
                                    │  观察结果     │
                                    └──────┬───────┘
                                           │
                              ┌────────────▼────────────┐
                              │  是否满足验收标准？      │
                              └───┬───────────────┬─────┘
                                  │ 是            │ 否
                          ┌───────▼──────┐   ┌───▼──────────┐
                          │ 产出物写入   │   │ 继续迭代？   │
                          │ 状态标记完成 │   └───┬──────────┘
                          └──────────────┘       │ 是     │ 否
                                             ┌───▼──┐ ┌──▼────────┐
                                             │ 继续 │ │ 标记失败  │
                                             └──────┘ └───────────┘

4.2 循环步数限制

安全机制：最大迭代步数。每个 Worker 执行单任务有最大步数限制（默认 50 步），超过则强制终止并标记为「步数超限失败」。防止模型死循环烧 token。这是从 Grok Build 的「无进展停滞检测」借鉴的简化版。

4.3 计划执行 vs 反应式执行

两种模式，由任务卡片指定：

模式适用场景特点
Plan-then-Execute
先计划后执行		 复杂任务、多步骤任务 先输出 plan.md，确认计划后再执行。计划可被 Centurion 或 Review Bee 审查。
ReAct
边想边干		 简单任务、探索性任务 经典 ReAct 模式，一步一想一干。适合不确定路径的任务。

借鉴 Grok Build 的 plan.md 契约思想：Plan-then-Execute 模式下，计划文件只规定 OUTCOME（验收标准），不规定 HOW（具体步骤）。Worker 在执行中可以调整路径，只要最终产出符合验收。

五、工具系统（Tool System）

5.1 工具分层

层级工具类型来源说明
L0 底座工具 文件读写、目录操作、shell 命令、web 搜索 nanobot 原生 所有 Worker 默认加载，不可禁用
L1 MCP 工具 任何 MCP server 提供的工具 nanobot MCP 客户端 通过配置 MCP server 动态接入。如：代码知识图、数据库查询、API 调用等
L2 技能包工具 特定领域的工具集合 新增 技能包机制 Python 模块形式，封装一组相关工具和提示词。如：文献整理技能包、代码审查技能包、数据处理技能包
L3 蜂群协作工具 领任务、交任务、查知识图、发通知等 新增 File-as-Bus 接口 Worker 与蜂群其他部分交互的工具

5.2 技能包（Skill Pack）机制

这是实现半异构的核心。所有 Worker 共享 L0+L1+L3，差异在 L2 技能包。

技能包结构

skill_packs/
├── literature_review/        # 文献整理技能包
│   ├── __init__.py
│   ├── tools.py              # 工具定义
│   ├── prompts/              # 提示词模板
│   │   ├── summarize.txt
│   │   └── classify.txt
│   └── manifest.yaml         # 技能包清单
├── data_analysis/            # 数据分析技能包
│   ├── __init__.py
│   ├── tools.py
│   └── manifest.yaml
└── code_review/              # 代码审查技能包
    ├── __init__.py
    ├── tools.py
    └── manifest.yaml

manifest.yaml 示例

name: literature_review
version: "0.1.0"
description: "文献整理与分析技能包"
tools:
  - search_papers
  - extract_metadata
  - summarize_paper
  - classify_paper
system_prompt_addon: |
  你擅长学术文献整理。你可以搜索论文、提取元数据、生成摘要、
  按研究主题分类。输出格式遵循学术规范。
dependencies:
  - name: "pdf"
    type: "mcp_server"
    config: { ... }

Worker 角色定义

# worker_profiles/researcher.yaml
name: "researcher"
description: "文献研究型 Worker"
skill_packs:
  - literature_review
  - data_analysis
mcp_servers:
  - arxiv
  - semantic_scholar
llm_config:
  model: "gpt-4o"
  temperature: 0.3
max_steps: 100

不同的 Worker Profile 加载不同的技能包 + MCP server，形成半异构的蜂群工蜂。

5.3 MCP 工具接入

完全复用 nanobot 的 MCP 客户端。Worker 启动时，根据配置自动启动并连接所有 MCP server，将其工具注册到工具管理器中。

设计决策：MCP 是蜂群工具扩展的标准接口。任何外部能力（数据库、API、专有工具）都通过 MCP 接入，而不是写死在代码里。这样 Worker 的工具扩展不需要改核心代码。

六、记忆系统（借鉴 hermes）

6.1 四层记忆架构

借鉴 hermes-agent 的记忆设计，但简化适配 Worker-Bee 的任务执行场景：

层级记忆类型存储位置生命周期用途
L1 工作记忆 当前任务的上下文 LLM 上下文窗口 单次任务内 任务执行中的即时思考
L2 情景记忆
（Episodic）		 任务执行过程的完整记录 任务产出目录 + 归档 永久 回溯执行过程、复盘、调试
L3 语义记忆
（Semantic）		 从任务中提炼的知识 SQLite 知识图谱 永久 + 可更新 跨任务知识复用、推理、关联
L4 程序记忆
（Procedural）		 工作流程和方法论 技能包 + system prompt 技能包版本管理 做事的方法、步骤、规范

6.2 上下文压缩机制

借鉴 hermes 的 4 阶段上下文压缩，但简化为 2 阶段：

  1. 实时摘要：当对话历史超过阈值（如 80% 上下文窗口），自动对较早的轮次做摘要，保留关键信息但压缩 token
  2. 任务结束归档：任务完成后，生成「任务元数据 + 核心结论 + 关键产出索引」，存入语义记忆（知识图谱）

不做 hermes 那么复杂的分层压缩是因为：Worker 任务通常短平快，单任务不需要那么多层压缩。跨任务的知识复用靠知识图谱，不靠上下文压缩。

6.3 记忆写入触发点

  • 任务开始：创建情景记忆条目，记录任务卡片内容
  • 每步执行：工具调用和结果自动记入情景记忆
  • 任务完成：生成任务总结，抽取知识节点和关系，写入语义记忆（知识图谱）
  • 任务失败：记录失败原因和教训，写入语义记忆的「失败案例」节点

6.4 记忆查询

  • 任务内查询：直接从工作记忆（上下文）取
  • 历史任务查询：通过知识图谱的语义搜索，找相关的历史任务和结论
  • 影响范围查询：通过 Blast Radius 算法，查某个产出变更会影响哪些任务

七、File-as-Bus 接口协议

7.1 命名规范即协议

核心设计：文件系统就是消息总线。所有 Bee 之间不直接通信，全部通过读写约定路径的文件来协作。文件名和目录结构就是协议。没有 broker、没有队列、没有 socket。

7.3 任务卡片格式

# task.yaml
task_id: T20260718_001
type: research
title: "综述智能体自组织相关文献"
priority: high
assigned_to: null           # null = 公开竞标，指定 worker_id = 指定派发
required_skills:            # 需要的技能标签
  - literature_review
  - data_analysis

# 任务目标
goal: |
  搜索并整理近3年关于"多智能体自组织"的核心论文，
  生成一份结构化的文献综述清单。

# 验收标准（最关键！）
acceptance_criteria:
  - 至少包含 30 篇相关论文
  - 每篇论文包含：标题、作者、年份、期刊、核心贡献
  - 按研究方向分类（至少 3 个类别）
  - 输出格式为 CSV + Markdown 摘要

# 输入资源
inputs:
  - type: file
    path: "artifacts/documents/research_topics.md"
  - type: knowledge_graph_query
    query: "已有的相关文献节点"

# 产出物规格
outputs:
  - name: paper_list
    type: csv
    path: "artifacts/data/paper_list_T20260718_001.csv"
  - name: summary
    type: markdown
    path: "artifacts/documents/literature_summary_T20260718_001.md"

# 执行约束
constraints:
  max_steps: 100
  mode: react          # react | plan_then_execute
  llm_model: null      # null = 使用默认
  deadline: null       # null = 无截止时间

# 溯源
created_by: C20260718_001   # 创建者（Centurion ID）
created_at: 2026-07-18T14:30:00Z
parent_task: null           # 父任务（如果是子任务）
project: swarm_arch_research

7.4 Worker 领任务的流程

  1. 扫描 task_pool/：找自己能做的任务（匹配 required_skills）
  2. 尝试领取：将任务文件移动到 in_progress/{task_id}/，并创建 worker_lock 文件（写入自己的 worker_id + 时间戳）
  3. 原子性保障：文件移动操作在同一文件系统上是原子的。谁先移动成功谁领到任务，天然避免竞争。
  4. 执行任务：在 in_progress/{task_id}/workspace/ 下干活
  5. 完成/失败：移动到 done/ 或 failed/
为什么不用数据库做任务队列？因为 File-as-Bus 的哲学是：文件系统就是最可靠的消息队列。mv 操作是原子的，目录列表就是队列状态，文件内容就是消息体。简单、可观测、可手动干预。出了问题直接看目录、改文件，不需要进数据库。

7.5 状态更新

Worker 在执行过程中，定期更新 status.json：

{
  "task_id": "T20260718_001",
  "worker_id": "W_researcher_03",
  "status": "in_progress",      # pending | in_progress | done | failed | blocked
  "current_step": 15,
  "max_steps": 100,
  "progress_description": "已搜索到 45 篇论文，正在分类整理",
  "last_heartbeat": "2026-07-18T14:45:23Z",
  "artifacts_created": [
    "workspace/raw_papers.json"
  ],
  "errors": []
}

Centurion 和 PM 可以通过读这个文件了解进度。不需要轮询 API，看文件就行。

八、生命周期与状态机

8.1 任务状态机

借鉴 Grok Build 的纯状态机 + 安全降级思想：

                    ┌─────────────┐
                    │   pending   │  （在 task_pool 里）
                    └──────┬──────┘
                           │ 领取成功
                    ┌──────▼──────┐
                    │ in_progress │  （在 in_progress/ 里）
                    └──┬───┬───┬──┘
                       │   │   │
           验收通过 ────┘   │   └──── 遇到阻塞
                   ┌──────┐ │ ┌───────┐
                   │ done │ │ │blocked│
                   └──────┘ │ └───┬───┘
                       │   │     │
              审查通过 │   │失败  │ 依赖就绪
                       │   │     │
                       │ ┌─▼─────▼──┐
                       │ │  retry?  │
                       │ └─────┬────┘
                       │       │
                       │    是 │  否
                       │       │
                       │    ┌──▼──────┐
                       │    │ failed  │  （在 failed/ 里）
                       │    └─────────┘
                       │
                  ┌────▼─────┐
                  │ archived │  （知识沉淀后归档）
                  └──────────┘

8.2 安全降级

借鉴 Grok Build 的「未知状态自动降级到最安全状态」：

  • Worker 崩溃重启后，发现自己有锁但状态不完整 → 降级到 blocked，不强行继续，等 Centurion 处理
  • 连续 N 步无进展（输出相同的 gap 描述）→ 自动暂停，标记为 blocked，避免死循环烧 token
  • 模型调用连续失败 → 降级到更便宜的模型，或标记失败

8.3 Worker 自身的生命周期

  1. 启动：加载配置 → 加载技能包 → 连接 MCP servers → 注册到 registry/workers/
  2. 工作：循环领任务 → 执行 → 交任务 → 休息（可配置空闲睡眠时间）
  3. 闲置：长时间没任务 → 进入低功耗模式，降低心跳频率
  4. 关闭：收到关闭信号 → 完成当前任务（或保存进度）→ 注销 → 退出

九、三阶段流水线集成（借鉴 Grok Build）

9.1 Grok Build 的 Goal Mode 启发

Grok Build 的 Plan → Build → Review 三阶段闭环，映射到蜂群就是三层 Bee 的天然分工：

Grok Build 角色蜂群对应职责失败语义
PlannerPM Bee把想法变成可执行的方案Fail-Closed（规划错了宁可不做）
WorkerWorker Bee执行具体任务—
Skeptic PanelReview Bee / Centurion并行审查产出物质量Fail-Open（多一个审查视角没坏处）
StrategistArchitect Bee卡壳时提供策略指导Fail-Open（策略建议多了无害）
SummarizerReporter Bee生成最终报告Fail-Open（总结差点不影响核心产出）

9.2 Worker-Bee 层面的三阶段微观循环

不仅宏观上三层 Bee 对应三阶段，单个 Worker 执行任务时也有微观的三阶段：

  1. Plan（微观规划）：理解任务 → 制定步骤 → （可选）写 plan.md
  2. Build（执行构建）：调用工具 → 产出结果
  3. Review（自我检查）：对照验收标准自查 → 不通过则返工

但 Worker 的自我审查是「浅审查」，只对照验收标准逐项核对。深度审查（质量、一致性、架构合理性）由专门的 Review Bee 做。

9.3 验收标准驱动

核心理念：验收标准就是契约。任务卡片上的 acceptance_criteria 是 Worker 唯一的「对错」判断依据。达标就交活，不达标就继续干。验收标准写不清楚的任务，Worker 有权不接（退回给 Centurion 重写）。

这直接借鉴 Grok Build 的 plan.md 契约思想：只规定 OUTCOME，不规定 HOW。Worker 用什么路径、什么工具、什么顺序达到验收标准，是 Worker 自己的事。

10.2 Worker 怎么用知识图

  1. 任务开始时：查询知识图，找相关的历史任务、已有知识、相关产出物，作为上下文输入
  2. 任务执行中：遇到不确定的概念 → 查知识图 → 补充上下文
  3. 任务完成后：从产出物中抽取知识节点和关系 → 写入知识图
  4. 影响分析：如果修改了某个产出物 → 用 Blast Radius 算影响范围 → 通知相关任务

10.3 知识图作为 L3 蜂群协作工具

知识图查询能力以工具的形式提供给 Worker：

  • graph_search(keywords)：语义搜索，找相关节点
  • graph_blast_radius(node_id, direction)：影响范围分析
  • graph_add_node(type, name, properties)：添加知识节点
  • graph_add_edge(from, to, relation, confidence)：添加关系边
  • graph_get_neighbors(node_id, depth)：取邻居节点

第一阶段只实现 search + blast_radius + get_neighbors 三个读操作。写操作先不开放给 Worker，由专门的知识整理 Bee 负责，防止写乱。

十一、半异构与技能包机制

11.1 为什么是半异构

设计决策：半异构，不是全同也不是全异。全同 Worker 太死板，全异 Worker 维护成本爆炸。半异构取中间：同底座（agent循环+工具框架+记忆系统），异在技能包和配置。

11.2 Worker Profile

每种 Worker 角色对应一个 profile 文件，定义加载哪些技能包、连接哪些 MCP server、用什么模型、步数限制等。

典型的 Worker 角色（逐步扩展）：

角色核心技能包典型任务
Researcher文献检索、信息整理、数据处理文献综述、资料收集、市场调研
Writer文档撰写、格式排版、内容编辑写报告、写文档、润色文稿
Coder代码编写、代码审查、测试写代码、改 bug、code review
Analyst数据分析、可视化、统计建模数据清洗、报表生成、建模分析
Reviewer质量检查、逻辑审查、格式校验审查产出物、验收任务、找漏洞

11.3 技能包开发规范

  1. 每个技能包是一个独立 Python 包
  2. 必须有 manifest.yaml 声明工具、依赖、系统提示词附加段
  3. 工具函数必须有完整的类型标注和 docstring（自动作为 tool schema）
  4. 技能包版本化，向后兼容
  5. 技能包可以依赖 MCP server（在 manifest 中声明）

十二、配置与多模型后端

12.1 配置层级

配置优先级（高 → 低）：
1. 任务卡片中的 constraints（单次任务覆盖）
2. Worker profile（角色级默认）
3. 蜂群全局配置（整个蜂群默认）
4. 环境变量（机器级）

12.2 多模型策略

复用 nanobot 的多模型后端，加上蜂群特有的模型路由策略：

  • 默认模型：Worker profile 指定（如 GPT-4o）
  • 降级模型：主模型不可用时自动降级到备选（如 Claude 3.5 Sonnet）
  • 工具调用专用模型：复杂 tool calling 用更强的模型，简单推理用便宜模型
  • 成本感知：可配置 token 预算，快超支时降级到便宜模型

十五、风险与开放问题

15.1 技术风险

风险概率影响缓解措施
nanobot 代码提取后跑不起来 低 中 边提取边测，每提取一个模块就跑测试。实在不行就用 nanobot 作为依赖 import，不提取
File-as-Bus 原子性问题 低 中 同一文件系统内 mv 是原子的。跨文件系统的话用锁文件 + 重试机制
知识图抽取质量差 中 中 置信度分级 + 人工/Review Bee 审核。低置信度的边默认不参与推理
Worker 死循环烧 token 中 中 最大步数限制 + 无进展检测 + token 预算 + 降级模型
SQLite 性能瓶颈 低 低-中 WAL 模式 + 合理索引。真到瓶颈了再考虑换图数据库。第一阶段绝对够用

15.2 开放问题

  1. Worker 数量：第一阶段是单 Worker 还是多 Worker 并发？多 Worker 的话竞争和冲突处理要提前设计。
  2. 任务调度策略：Centurion 分配还是 Worker 竞标？第一阶段建议 Worker 竞标（pull 模式），简单。
  3. 知识图写入权限：Worker 能不能直接写知识图？还是必须通过专门的知识整理 Bee？第一阶段建议只读，由 Review Bee 或专门的知识 Bee 写。
  4. 人工干预路径：出了问题人怎么介入？直接改文件？还是有 CLI 工具？
  5. 和你的协作方式：你怎么给蜂群发任务？通过 CLI、通过文件、通过某个 Bee 入口？
